Prototype Plan — Competitor Overnight Brief

3-day build (Wed–Fri), demo Saturday. Written March 25, 2026.

An agent you launch before bed that does real multi-step work overnight, and greets you with a morning briefing card showing what it did, what it decided, and what it needs you for.

Demo Scenario: Why Competitor Monitoring

Picking the right demo scenario matters more than the architecture. The “Competitor Overnight Brief” was chosen because it satisfies all three ICPs:

  • Cyril (ops, Level 3 delegation): “Here’s the outcome I want, here’s roughly how, use your judgment.” A competitor brief is exactly this.
  • Kautuk (technical founder, 80% done): The overnight brief gets him 80% of the way. He reads it, makes two edits, forwards it to the team. Unblocked before his first coffee.
  • Deepanshi (designer, memory is #1): The agent remembers what it monitored last time, what the founder cares about, what was already known. It doesn’t repeat itself.

Why NOT Other Scenarios

  • Lead enrichment/outreach: Requires actual email sending. Irreversible actions in a demo = trust disaster.
  • Content pipeline: Output quality is subjective. Audience debates “quality” instead of appreciating agent behavior.
  • Scheduling/coordination: Requires multiple real calendar accounts. Setup overhead kills the demo.
  • Financial reporting: Nobody wants to show their Stripe dashboard on a projector.

Competitor monitoring is perfect because: (a) the output is a document — reviewable and reversible, (b) the value is obvious to any founder, (c) you can show real data from real companies, (d) the “overnight” framing is natural.

The Demo Flow

Setup (Before Bed)

  1. Open Felix. Swipe to the “Launch Agent” card.
  2. Natural language input: “Monitor [competitor] — track their pricing page, blog, changelog, job postings, and any news. Compare to our positioning on [feature area]. Tell me what changed and what I should do about it.”
  3. Agent asks one clarifying question (demonstrates it’s not blindly executing): “Should I also check their social media and community forums, or just official channels?”
  4. User answers. Taps “Launch.” Goes to sleep.

Overnight (Time-Lapse / Replay)

  1. Agent breaks the task into steps (visible in execution log).
  2. Scrapes competitor pricing page, blog, changelog, careers page.
  3. Searches for recent news mentions and social signals.
  4. Compares findings against stored context (last known competitor state).
  5. Identifies 2–3 meaningful changes. Drafts analysis with recommended actions.
  6. Flags one item it’s uncertain about — marks it for morning review rather than guessing.

Morning (The Money Shot)

  1. Felix notification: “Your overnight brief is ready.”
  2. Swipeable card with summary: “3 changes detected. 1 needs your input.”
  3. Tap to expand: full brief with sections for each change, what it means, suggested response.
  4. One card says: “I found a new job posting for ‘Head of Enterprise Sales’ — this could signal an upmarket move, but I’m not confident. Want me to dig deeper?” (The trust moment — the agent admitting uncertainty.)
  5. User swipes to approve or dismiss.

Architecture

Principle: Fake nothing, simplify everything. Real data, not canned responses. But the architecture can be radically simplified.

Key Components

  • Felix UI: If iOS card infrastructure exists, use it. Otherwise, a polished web mockup shown on a phone browser. Three screens: launch input, execution log, morning brief card.
  • Launch API: Single POST endpoint. Accepts task description + user context. Returns a job ID.
  • Agent Orchestrator: Plain Python + Claude API. No LangChain, no CrewAI, no abstractions. Hardcoded pipeline for competitor monitoring. See the skeleton code.
  • Context Store: A JSON file per user. Company description, competitor last-known state, previous briefs. No database, no vector store.
  • Web Fetch: HTTP requests + Jina Reader as fallback for JS-rendered pages. Target 3–4 specific pages.
  • News/Social: One API call to a news aggregator (Google News RSS or NewsAPI.org). Skip social for v1.

What to Skip

  • Authentication/user accounts — hardcode the demo user
  • Real push notifications — simulate with polling or manual trigger
  • General-purpose agent routing — this agent does ONE thing
  • Error recovery — if a fetch fails, log it and move on
  • Concurrent agents — one agent, one task, one user
  • Agent frameworks — plain Python + Claude API calls

3-Day Build Plan

Day 3 (Wednesday): Backend Pipeline

  • Get the orchestrator working end-to-end on a single competitor.
  • Input: competitor name + what to monitor. Output: a JSON brief.
  • Test against a real company (well-known startup with public pricing page, active blog, changelog).
  • By end of day: python run_agent.py "monitor Linear" produces a useful brief.

Day 4 (Thursday): UI + Context

  • Build the three UI screens (launch, execution log, morning brief).
  • Wire the API endpoint. Launch a task from the UI, see the brief appear.
  • Add context store: save competitor’s current state so next run shows diffs.
  • By end of day: Run the agent twice to demonstrate “here’s what changed since last time.”

Day 5 (Friday): Polish + Rehearse

  • Add uncertainty flagging (agent marks items it’s unsure about).
  • Polish the morning brief card — this is the screenshot people will remember.
  • Write the execution log so it tells a story (timestamps, step descriptions, not raw API dumps).
  • Rehearse the demo 3–4 times. Time it. Cut anything taking longer than 30 seconds to explain.
  • Pre-run the agent the night before the demo so you have a real overnight brief to show Saturday morning.

Making the Demo Compelling

The Narrative Arc

Don’t demo features. Tell a story:

“It’s 11 PM. You’ve been in back-to-back meetings all day. Your investor just asked how you’re positioned against [competitor]. You have no idea what they shipped this month. You could spend the next two hours researching, or…”

[Launch the agent. Go to sleep. Time-lapse the execution log. Wake up.]

“Three things changed. One of them matters. Here’s what to do about it. And here’s what the agent wasn’t sure about. It didn’t guess. It asked you.”

Under 5 minutes.

Moves That Land

  1. Use a real competitor the audience knows. The brief should contain real information that makes people go “wait, did that actually change?”
  2. Show the execution log briefly, don’t dwell on it. Scroll through for 5 seconds. The morning brief is the hero.
  3. The uncertainty moment is the trust moment. When the agent says “I’m not sure about this” — that’s when the audience feels safe.
  4. End with the counter-factual. “Without this, you’d have spent 90 minutes tonight doing what the agent did in 12 minutes while you slept.”
  5. Have a live run ready, but also have a pre-baked run. If the live demo gods are cruel, show the pre-baked result.

What NOT to Do

  • Don’t show the architecture diagram at demo day.
  • Don’t demo error handling.
  • Don’t claim it’s general-purpose. Say “we built this for competitor monitoring as the first use case.”
  • Don’t mention LangChain, vector databases, RAG, or any AI buzzword.

Risks

Risk 1: The LLM Produces a Bad Brief

Likelihood: Medium. Claude is good at synthesis, but if scraped data is noisy or the competitor hasn’t changed anything, the brief could be generic.

  • Pick a competitor that’s actively shipping. Check their blog before choosing.
  • Seed the context store with “last known state” from a week ago so there ARE diffs to find.
  • Spend real time tuning the synthesis prompt — it’s the most important 50 lines of code.
  • Have a pre-baked brief ready as backup.

Risk 2: Web Scraping Fails

Likelihood: Medium-High. Cloudflare and JS rendering will cause trouble.

  • Test target pages early on Day 3. If blocked, pick a different competitor.
  • Use Jina Reader (reader.jina.ai) as fallback — handles JS, returns clean markdown.
  • Pre-cache raw HTML from successful test fetches as a fallback.

Risk 3: The Demo Takes Too Long

Likelihood: High if you don’t rehearse. Agent execution can take 2–5 minutes.

  • Pre-run the agent. Show the result. Optionally show a live launch and cut to “we’ll check back.”
  • Use the execution log as visual progress during live execution.
  • Hard cap the demo at 5 minutes. Practice until you can do it in 4.

Risk 4: The “So What” Problem

Likelihood: Low if you nail the narrative. Risk is people thinking “I could just set up a Google Alert.”

  • Emphasize judgment, not fetching. Google Alerts says a page changed. This agent says what it MEANS and what to DO.
  • Show context awareness: “It knows your product does X, so when the competitor adds X, it flags that.”
  • Show uncertainty handling: “Google Alerts can’t tell you it’s unsure.”

Risk 5: Scope Creep

Likelihood: Very high. Someone will want Slack integration, multi-competitor support, a dashboard…

  • Scope is: one competitor, one user, one overnight run, one morning brief card. Write it on the whiteboard.
  • Every feature idea goes on a “post-demo” list.
  • Decision filter: “Does this make the 5-minute demo better?” If no, it waits.

What “Good” Looks Like on Saturday

The demo is good if:

  1. A founder in the audience thinks: “I would use this tonight.”
  2. The morning brief contains real, accurate, non-obvious information about a real company.
  3. The agent demonstrates judgment (interpretation, not just data fetching).
  4. The agent demonstrates appropriate humility (flagging uncertainty).
  5. The whole thing takes under 5 minutes.
  6. Someone asks “can it do [other task]?” — they bought the concept and are thinking about expansion.

The demo fails if:

  1. The output looks like a ChatGPT prompt response (no multi-step reasoning, no context awareness, no memory).
  2. The audience focuses on technical problems rather than value proposition.
  3. It takes more than 2 minutes to explain what the product does before showing it.

Post-Demo Q&A Prep

  • “How is this different from ChatGPT?” — ChatGPT doesn’t run while you sleep, doesn’t remember last time, doesn’t know your competitive position, doesn’t stop itself when unsure. This is an agent, not a chatbot.
  • “What other tasks could it do?” — Architecture generalizes. Competitor monitoring was first because it’s the clearest overnight task. Lead research, content pipelines, and customer feedback synthesis are next.
  • “What about hallucination?” — Agent only synthesizes from data it actually fetched. When uncertain, it tells you. Design choice, not limitation.
  • “What if it does something wrong overnight?” — Nothing irreversible. It produces a document. Worst case is a bad brief you ignore. Approval gates for irreversible actions come later.
