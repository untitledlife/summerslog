Part I: SOTA Quantization Landscape

The Quick Map

FrontierState (Apr 2026)

Weight-Only PTQ

Foundational (solved)

  • GPTQ (ICLR 2023): Hessian-based one-shot, W4, <1 ppl loss. Industry default.
  • AWQ (MLSys 2024): Activation-aware channel scaling, W4, <0.5 ppl. Preferred for edge.
  • HQQ: No calibration data needed. Fast minutes-scale quantization.

Extreme compression (frontier)

  • QuIP# (ICML 2024): E8 lattice codebooks + Hadamard rotations. LLaMA-2-70B at 2-bit: 3.93 ppl (vs 3.32 FP16).
  • AQLM (ICML 2024): Multi-codebook additive VQ. Competitive at 2-bit, functional at ~1-bit.
  • QTIP (NeurIPS 2024): Trellis-coded quantization. Sub-2-bit, beats QuIP#/AQLM.
  • SqueezeLLM (ICML 2024): Dense-and-sparse decomposition, strong at W3.
  • OmniQuant (ICLR 2024): Learnable clipping + transforms. Bridges PTQ↔QAT.
  • SpQR (ICML 2024): Sparse outliers at FP16, rest at 3-4 bit.
  • BiLLM (ICML 2024): Binary (1.1-bit) via Hessian-aware binarization. Significant quality gap.

Weight + Activation Quantization

  • SmoothQuant (ICML 2023): Per-channel scaling, W8A8, near-lossless. Industry standard.
  • Atom (MLSys 2024): Mixed-precision W4A4 co-designed with kernels. 7.7x throughput.
  • QoQ/QServe (MLSys 2025): W4A8KV4. Practical production system. 1.2-3.5x speedup.
  • QuaRot (NeurIPS 2024): Hadamard rotations for full W4A4KV4 inference.
  • SpinQuant (NeurIPS 2024, Meta): Learned optimal rotations, strictly better than QuaRot.
  • FlatQuant (NeurIPS 2024): Learned affine transforms to flatten loss landscape. SOTA W4A4.
  • SageAttention2 (2025): INT4/FP8 attention, ~3x over FlashAttention-2.

KV Cache Quantization

  • KIVI (ICML 2024): Tuning-free asymmetric 2-bit. Per-channel keys, per-token values. <0.1 ppl loss. Widely adopted.
  • KVQuant (NeurIPS 2024): Per-channel NUQ with codebooks. Enables 1M context on single A100.
  • GEAR (NeurIPS 2024): Low-rank error approximation + sparse correction. ~3.4x compression.
  • QJL (AAAI 2025): 1-bit JL projections. Approximate but fast.

TurboQuant (Google, ICLR 2026): Two-stage near-optimal VQ. PolarQuant (rotation + polar quant) + QJL (bias correction). Within 2.7x of Shannon bound. Lossless at 3.5 bits. 4x speedup vs FP16 attention kernels. JAX, no open-source yet. Reportedly deployed for Gemini. This is the new SOTA.

QAT

  • BitNet b1.58 (Microsoft 2024): Ternary {-1,0,+1} from scratch. Matches FP16 at 3B. Scaling to 13B+ in 2025-2026.
  • EfficientQAT (2024): Two-phase -- fast block-wise calibration + optional E2E fine-tune. SOTA at W2.
  • QLoRA (NeurIPS 2023): NF4 base + BF16 LoRA. Ubiquitous for fine-tuning.
  • LoftQ (ICLR 2024): Joint LoRA init + quantization. Better than QLoRA at 2-bit.
  • Native FP4/FP8 QAT (2025-2026): Emerging for Blackwell hardware.

Hardware-Aware

  • Marlin (2024): W4A16 CUDA kernel, ~3.87x (near-theoretical 4x) on A100. Gold standard.
  • Machete (vLLM, 2025): Marlin successor for Hopper, W4A8/W8A8, TMA support.
  • FLUTE (ICML 2025): Generic lookup-table kernel for any quantization format. Near-Marlin speed.
  • FP8 on Hopper: Mature, near-lossless, ~2x over FP16. Becoming default for serving.
  • FP4 on Blackwell: Early but promising. ~2x over FP8 theoretical. Software stack maturing.
  • llama.cpp/GGUF: Most deployed quantized runtime. CPU/Apple Silicon/GPU universal.
  • bitnet.cpp: CPU ternary inference, 1.4-5x speedup vs FP16 with 41-72% energy reduction.

What's NOT Solved (Gaps)

  1. Sub-2-bit PTQ without significant loss. QAT works at 1.58 bits (BitNet) but PTQ below 2 bits still degrades badly.
  2. The PTQ-QAT gap below 4 bits. Getting QAT-quality at PTQ-cost is the holy grail.
  3. W4A4 is not plug-and-play. Requires rotations, clipping, per-model tuning. Not like W4A16 which "just works."
  4. Quantization breaks reasoning. Multi-step CoT is 5-10% more sensitive than perplexity suggests. No principled fix.
  5. No universal calibration-free PTQ. KIVI/TurboQuant are cal-free for KV cache; HQQ for weights; nobody for W+A.
  6. MoE quantization is underexplored. Inactive experts tolerate lower precision -- nobody exploits this well.
  7. Multimodal quantization. Vision encoders, cross-attention, multi-modal projectors have different precision needs. No unified framework.
  8. Non-Transformer quantization. Mamba/RWKV have different bottlenecks (recurrent state vs KV cache). Transformer methods don't transfer directly.
  9. Dynamic per-token precision. Promising idea, no deployable solution.
  10. Kernel fragmentation. Every format needs custom CUDA kernels. FLUTE helps but early.

Where It's Heading

  • FP8 becoming default. FP4 at the frontier.
  • BitNet at 70B+ -- if quality holds, changes inference economics fundamentally.
  • Custom silicon for ternary ops emerging.
  • Architectures designed to be quantization-friendly (not post-hoc).
  • Compound compression: quant + pruning + distillation + NAS jointly.

Part II: Unsloth Technical Teardown

What It Is

A monkey-patching framework (~90K LOC across unsloth + unsloth-zoo) that intercepts HuggingFace Transformers models at load time and replaces forward/backward passes with optimized versions. Founded by Daniel and Michael Han-Chen.

Six Optimization Layers

1. Custom Triton Kernels unsloth/kernels/

  • Cross-Entropy Loss (463 LOC): Forward computes logsumexp in single fused kernel. Backward writes gradients in-place on logits tensor. Chunked approach for large vocabs (>65K). Major memory win.
  • RMSNorm (340 LOC): Fused forward+backward. Stores only inverse variance scalar, not full intermediate. Gemma-specific (W+1) variant.
  • RoPE (466 LOC): Fused Q+K rotation in single kernel launch. Groups heads (ROPE_GROUP_SIZE=4). Backward = negate sin and re-apply.
  • SwiGLU (143 LOC): Fused activation + gate. Backward reuses input buffers for derivatives -- saves 3 intermediate tensor allocations.
  • MoE Grouped GEMM: For Qwen3-MoE, DeepSeek-V3, etc. TMA descriptors for H100+. AGPL licensed.

2. Custom LoRA Backward Passes fast_lora.py (730 LOC) -- the biggest optimization

  • LoRA_QKV: Fused backward for Q/K/V projections. Manually implements chain rule. Dequantizes 4-bit weight once, does matmul, adds LoRA gradient via addmm_ (in-place). Avoids materializing intermediate activation tensors that PyTorch autograd would save.
  • LoRA_MLP: Same for gate/up/down MLP. Shares dequantized weight tensors.
  • LoRA_W: Generic single-projection backward (O projection).
  • Key insight: standard autograd saves input/output at each nn.Linear boundary. Unsloth's custom backward saves only X, e (gate), g (up), recomputes everything else.

3. Fused Cross-Entropy Without Materializing Logits

  • Computes loss directly from hidden states. Never allocates the full [batch, seq_len, vocab_size] tensor. For 128K vocab + seq_len=4096, saves ~2GB VRAM per forward pass.
  • Uses Apple's cut_cross_entropy library optionally.

4. Smart Gradient Checkpointing (3 modes)

  • CPU offloading: Async transfer to pinned CPU memory via CUDA streams. 200 pre-allocated 128KB buffers. Skips last layer. Falls back to standard for seq < 512.
  • Standard: Clean reimplementation.
  • Adaptive: Heuristics-based mode selection.

5. torch.compile AST Transforms compiler.py (4285 LOC)

  • AST-parses HuggingFace model source code
  • Source-level transforms: removes causal mask creation, replaces SDPA calls, fuses ops
  • Wraps with torch.compile(dynamic=True) with tuned options
  • Disk-cached compiled modules
  • Generic path for models without hand-optimized kernels

6. Tiled MLP

  • Chunks sequence dimension into tiles fitting a target GPU memory budget (default 0.5GB). Bounds peak memory for large intermediate dims (especially MoE).

Supported Models

Optimized (hand-written kernels): Llama 2/3/3.1/3.2/4, Mistral, Qwen2/3/3-MoE, Gemma 1/2/3/3N/4, Cohere, Granite, Falcon-H1, DeepSeek-V3/R1, GLM4-MoE, Phi-3.

Generic (torch.compile path): Any AutoModelForCausalLM or AutoModelForImageTextToText.

What They Don't Do

  • No custom attention kernel. Uses PyTorch SDPA / Flash Attention / xformers.
  • No multi-GPU/distributed training. Single-GPU focused. FSDP/DeepSpeed support is basic.
  • No FP8 training kernels. FP8 loading/inference only.
  • No kernel autotuning. Fixed block sizes via simple heuristics.
  • No long-context (ring attention, sequence parallelism).
  • No speculative decoding.

Part III: Gaps & Opportunities (What CE Could Build)

High-Impact, Buildable in 2-3 Months

  1. Multi-GPU LoRA training with fused kernels. Unsloth's biggest gap. Combine their kernel-level optimizations with FSDP/DeepSpeed. Nobody has this. A team that nails LoRA + FSDP + gradient checkpointing + custom Triton kernels on multi-GPU would have a differentiated product.
  2. FP8 LoRA training. No good open-source solution exists. Train LoRA adapters in FP8 forward / FP16 backward. Blackwell's FP4 adds another frontier. First mover advantage.
  3. Activation compression instead of CPU offloading. Quantize activations to INT8/FP8 in-place for gradient checkpointing rather than offloading to CPU. Halves memory without PCIe bandwidth cost. Straightforward to implement.
  4. Kernel autotuning. Unsloth's Triton kernels have fixed block sizes. Hardware-specific autotuning (like FlashAttention's approach) could yield 15-30% additional gains. Generic framework that generates optimal kernels per model config + hardware.
  5. Non-Transformer fine-tuning optimization. Nobody has Unsloth-equivalent for RWKV/Mamba/SSM models. Different bottleneck (recurrent state), different optimization opportunities. Directly relevant to CE's RWKV-BD3 work.

Research Angles (Publishable)

  1. Quantization for diffusion LMs. Nobody has studied how quantization affects discrete diffusion models (MDLM, BD3-LM, drLM). Different dynamics than AR -- denoising is iterative, error compounds differently. Original CE territory.
  2. MoE-aware mixed-precision. Adaptively allocate precision per expert based on routing frequency and sensitivity. Active experts at higher precision, inactive at 2-bit. Simple idea, no good paper on it.
  3. Quantization sensitivity of reasoning. 2026 evidence says CoT is more sensitive to quantization. Why? Is it the long-range dependencies? The compositional structure? A principled study would be publishable.
  4. Recurrent state quantization for SSMs. Analogous to KV cache quantization but for Mamba/RWKV state. Different distribution properties, different optimal strategies. Unexplored.
  5. TurboQuant for weights (not just KV cache). TurboQuant's PolarQuant + QJL composition is proven near-optimal for inner products. Could this be applied to weight quantization? The math might transfer.

What's Reproducible vs What's Hard

Reproducible in weeks (2-3 researchers)

  • Core Triton kernels (RMSNorm, RoPE, SwiGLU, CE) -- each 100-400 LOC, standard Triton
  • Custom LoRA backward -- ~700 LOC of manual chain rule
  • Fused CE without logit materialization -- existing technique (cut_cross_entropy)
  • Smart gradient checkpointing -- standard engineering

Hard to reproduce

  • Model coverage breadth (1466 LOC mapper, dozens of temp patches for new architectures) -- constant maintenance burden
  • torch.compile AST pipeline (4300 LOC) -- genuinely complex, parsing arbitrary HF model source
  • Ecosystem (pre-quantized models on HF, TRL/PEFT integration, vLLM)

The Honest Assessment

Unsloth's individual optimizations are not deeply novel. Kernel fusion, buffer reuse, avoiding materialization, CPU offloading -- all standard GPU optimization techniques. What makes it work is the integration + zero-code-change UX.

The moat is not any single kernel -- it's the breadth of model support and the monkey-patching UX that makes it "just work" with existing HF code. Competing on that axis means an ongoing maintenance race.

Where to compete instead: Multi-GPU (they're weakest), FP8 training (nobody has it), non-Transformer architectures (original territory), and research publications that become the reference implementation (like FlashAttention did for attention kernels).
